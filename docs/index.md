# context Provider

The **context** provider gives you a structured, composable way to derive consistent resource IDs and tags from a shared context stack.

A context stack is an ordered list of named _namespace_ entries that represent organisational layers (e.g. environment, region, team). A `context_label` data source reads the stack and computes a canonical `id` and a default `tags` map for a specific resource. This removes the need to hand-craft resource names in every module.

Key features:

- **Composable namespaces** – stack as many `context_namespace` data sources as you need.
- **Flexible ID shaping** – control casing (`id_casing`), add a prefix (`id_prefix`), and optionally include the resource type in the ID (`include_resource_type_in_id`).
- **JQ-powered mappers** – override or enrich `id` and `tags` with arbitrary JQ expressions, declared inline or loaded from a JSON file.

## Mappers

Mappers are JQ expressions that modify the context output, empowering complex uses cases, including:

- setting different tags or using a different format for the label IDs for specific resources  
  E.g. you use only the label name as ID, without namespaces, when the label's resource_type is "iam-permission":

  ```json
  [
    {
      "name": "IAMPermissionShortName",
      "function": "(.context.output.id = .stack[-1].name).context",
      "run_condition": ".vars.resource_type == \"iam-permission\""
    }
  ]
  ```

- store and use later on other attributes, besides `id` and `tags`, for more complex logic

Mappers are JSON objects with three attributes:

1. `name` - unique name identifier for the mapper  
   This allows overwritting a mapper's function or run condition, upper in the stack
2. `function` - the JQ expression that modifies the context  
   E.g.`(.context.outputs.tags += { Environment = .vars.environment }).context`  
   Currently this expresion must follow the pattern: `(.context.outputs.$key ...).context`  
   Available keys are:
    - `vars` - a map obtained from merging the `vars` defined at the provider level, with all `vars` in namespaces in the context stack
    - `stack` - the context stack (`context.stack` from `context_label` data source)
3. `run_condition` - a JQ expression that must evaluate to bool and determines if the mapper's function is executed to modify the context  
   This will have the same data available as the `function`

> ⚠️ **Mappers are evaluated in top-down order. That means mappers last in the context stack are evaluated first.**  
> E.g. a mapper that modifies the `id` attribute defined at the provider config level, takes precendence over mappers in namespaces modifying the `id`  
> This is done to allow modifying behavior of "blackbox" modules, allowing the user to change the behavior of a module that's using the context provider, without changing the module's source code.

## Example Usage

```terraform
terraform {
  required_providers {
    context = {
      source = "snapsoft-io/context"
    }
  }
}

# Provider-level defaults apply to every data source unless overridden at the
# namespace level.  All three ID-shaping attributes (id_casing, id_prefix,
# include_resource_type_in_id) can also be set individually on each
# context_namespace data source for finer-grained control.
provider "context" {
  id_casing = "kebab-case"

  vars = {
    organization = "acme"
    team         = "platform"
  }

  # Optional: load JQ-based mapper functions from an external JSON file.
  # Inline mappers can alternatively be declared with the `mappers` block.
  mappers_file_path = "provider-mappers.json"
}

# ── Build the context stack ───────────────────────────────────────────────────

data "context_namespace" "env" {
  name    = "production"
  context = { stack = [] }
}

data "context_namespace" "region" {
  name    = "eu-west-1"
  context = data.context_namespace.env.context
}

# ── Produce a resource label ──────────────────────────────────────────────────

data "context_label" "bucket" {
  name          = "assets"
  resource_type = "s3-bucket"
  context       = data.context_namespace.region.context
}

output "bucket_id" {
  value = data.context_label.bucket.id
  # => "production-eu-west-1-assets"
}

output "bucket_tags" {
  value = data.context_label.bucket.tags
  # => { Name = "production-eu-west-1-assets", Team = "platform", Organization = "acme" }
}
```

## Schema

### Optional

- `id_casing` (String) Casing style applied to generated IDs. Accepted values: `kebab-case` (default), `snake_case`, `camelCase`, `PascalCase`. Can be overridden per `context_namespace` data source.
- `id_prefix` (String) Fixed string prepended to every generated ID. Can be overridden per `context_namespace` data source.
- `include_resource_type_in_id` (Boolean) When `true`, the `resource_type` value from `context_label` is appended to the generated ID. Can be overridden per `context_namespace` data source.
- `mappers` (Attributes List) Inline JQ-based mapper functions applied when `context_label` evaluates the stack. Mutually exclusive with `mappers_file_path`. (see [below for nested schema](#nestedatt--mappers))
- `mappers_file_path` (String) Path to a JSON file containing mapper function definitions. Mutually exclusive with `mappers`.
- `vars` (Map of String) Provider-level variables available to all mappers. Namespace-level `vars` take precedence on key conflicts.

### Nested Schema for `mappers`

Required:

- `function` (String) JQ expression that receives `{context, stack, vars}` and must return a new `context` object.
- `name` (String) Human-readable name for the mapper, used in error messages.

Optional:

- `run_condition` (String) JQ expression that receives `{context, stack, vars}` and must return a boolean. The mapper runs only when this evaluates to `true`.
